# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

NailGun (NG) is a Python orchestration wrapper for the ISAX (Instruction Set Architecture Extension) integration flow. It coordinates a multi-stage pipeline for custom RISC-V core extensions:

**CoreDSL → Treenail (TN) → Longnail (LN) → SCAIE-V → extended RISC-V core → cocotb simulation → LibreLane**

## Build Commands

```bash
make gen_config    # Scan isaxes/ and cores/ to generate Kconfig menus
make menuconfig    # Interactive configuration UI (requires gen_config)
make build         # Run the full pipeline via dispatch.py
make ci            # CI mode: gen_ci_config + build (uses env vars for config)
make clean         # Remove build/
make mrproper      # Remove build/, outputs/, test_results/
```

The pipeline reads `.config` (generated by menuconfig or CI). To override: `CONFIG_PATH=path/to/.config make build`.

For CI, configuration is driven by environment variables mapped to Kconfig symbols via `gen_ci_config.py`.

### `make ci` Environment Variables

`make ci` runs `gen_ci_config.py` (env vars → `.config`) then `dispatch.py` (execute pipeline).

| Env Var | Effect |
|---|---|
| `CORE` | Select core (e.g. `CVA5`, `CVA6`) → sets `CORE_<name>=y` |
| `ISAXES` | Comma-separated ISAXes (e.g. `SBOX,SPARKLE`) → sets `ISAX_<name>_EN=y` |
| `SIM_ENABLE` | Enable simulation (`"y"`) |
| `TB_PATH` | Testbench source (`.cpp`, `.S`, `.yaml`) |
| `TB_EXPECTED_PATH` | Expected output file for comparison |
| `SIM_CYCLE_TIMEOUT` | Cycle limit for simulation (default 50000) |
| `SIM_ENABLE_ISS_LOCKSTEP` | Enable ISS lockstep verification |
| `NO_ISAX` | Use `NO_ISAX_ENTRY_POINT` — skip ISAX integration (baseline test) |
| `COREDSL_MLIR_ENTRY_POINT` | Use pre-compiled MLIR instead of CoreDSL |
| `MLIR_ENTRY_POINT_PATH` | Path to the MLIR file (with `COREDSL_MLIR_ENTRY_POINT`) |
| `SV_ENTRY_POINT_PATH` + `SV_ENTRY_POINT_ISAX_YAML_PATH` | Direct SV entry point |
| `CONFIG_PATH` / `OUTPUT_PATH` | Override config/output locations |
| `LN_ILP_SOLVER` | Select ILP solver → sets `LN_SOLVER_USE_<solver>=y` |
| `LN_CELL_LIBRARY` / `LN_CLOCK_PERIOD` | Longnail cell library and clock period |
| `LN_SCHED_ALGO_MS` / `LN_SCHED_ALGO_PA` | Enable scheduling algorithms |
| `LN_OPTY_OL2_MODEL` / `LN_OPTY_CUSTOM_MODEL_PATH` | Longnail optimization models |
| `LN_PREDEFINED_SOLUTION_SELECTION` | Skip interactive solution selection |
| `LN_ENABLE_DATAPATH_SPLIT` | Enable split-datapath sharing (requires `LN_SCHED_ALGO_MI`; legacy 1-op-per-slot behavior when off) |
| `LN_OPTIMIZE_SLOT_BINDINGS` / `LN_OPTIMIZE_SLOT_BINDINGS_TIMEOUT` | Run ILP slot-binding optimization (and its timeout) |
| `LN_VISUALIZE_MRT` | Print ASCII MRT before/after slot-binding optimization |
| `LN_SPLITOP_GEN_TESTBENCH` | Also generate cocotb testbenches for split-op modules |
| `ONLY_PATCH_CC` | Only build the dynamic-ISAX clang — skip HLS/SCAIE-V/sim (see ONLY_PATCH_CC section below) |
| `SCAIEV_DO_NOT_REBUILD` | Skip rebuilding SCAIE-V |
| `SIM_TB_COMPILE_FLAGS` | Extra compiler flags for testbench compilation |
| `SCV_INTERNAL_CONTEXTS_AMOUNT` | SCAIE-V multi-context register banking |
| `SIM_ISS_PREDEFINED_ISAXES` | Predefined ISS ISAX definitions (e.g. `zol`) |

Minimal single-test example:

```bash
CORE="CVA5" ISAXES="SBOX" SIM_ENABLE="y" TB_PATH="custom_tbs/sbox.cpp" TB_EXPECTED_PATH="custom_tbs/sbox_expected.txt" make ci
```

Any env var whose name matches a Kconfig symbol is also passed through 1:1 (see `gen_ci_config.py`).

### `ONLY_PATCH_CC` Mode

When `ONLY_PATCH_CC="y"`, the pipeline skips HLS, SCAIE-V, and simulation. The reduced flow is:

1. **Treenail**: CoreDSL → individual MLIR files (via `entrypoint.resolve_mlir_paths()`)
2. **Merge ISAXes**: `longnail.concat_mlir_files()` + `longnail.merge_isaxes()` — combines multiple ISAXes into one MLIR
3. **Analyze ISAXes**: `toolchain.run_analyze_isax()` runs the `analyze-isax` ShortNail pass on the merged MLIR to produce a structured YAML with instruction names, encoding masks, and field metadata. This YAML is consumed by the dynamic-ISAX clang at compile/disassembly time via `-misax-desc=<yaml>` (and `-mllvm=-isax-desc=<yaml>` for `llvm-objdump`).
4. **Build dynamic-ISAX clang**: `toolchain.prepare_dynamic_isax_llvm()` configures and builds the `deps/llvm_dynamic_isax` fork (LLVM 21) if not already built. Full Longnail HLS (scheduling, HW-gen) is not needed.
5. **Precompile picolibc** for all cores (when TB is `.cpp`)

Example (build dynamic-ISAX clang with all ISAXes):
```bash
ONLY_PATCH_CC="y" CORE="CVA5" ISAXES="SBOX,SPARKLE,AUTOINC" TB_PATH="custom_tbs/dummy.cpp" TB_EXPECTED_PATH="custom_tbs/dummy_expected.txt" make ci
```

### Integration Tests (`tools/run_integration_tests.py`)

Runs the full test matrix across cores and ISAXes:

1. **Init**: `make gen_config`, then a `make ci` with all ISAXes on CVA5 to generate MLIR files
2. **Sequential tests**: Tests that cannot be parallelized
3. **Merge MLIR**: Collects all `.mlir` files → merges into `ALL_ISAXES.mlir`
4. **Patch compilers**: Prepares clang with merged ISAX support
5. **Parallel tests**: Runs remaining tests concurrently (up to `cpu_count` threads, Scala cores capped at 8)

Per-core feature gating (`CoreFeature` flags: Memory, Decoupled, Control, RdRD, MultiContext, ISSLockstep) filters out unsupported test/core combinations. Supported cores: CVA5, CVA6, CVA6_DUAL, PICORV32, PICCOLO, ORCA, VEX_4S, VEX_5S, CV32E40X.

## CoreDSL Language

CoreDSL 2 (`.core_desc` files) is a domain-specific language for describing RISC-V ISA extensions. Reference: [CoreDSL 2 programmer's manual](https://github.com/Minres/CoreDSL/wiki/CoreDSL-2-programmer's-manual).

### Structure

A `.core_desc` file contains `InstructionSet` and/or `CoreDefinition` blocks. Instruction sets support inheritance via `extends`. Each contains:

- **`architectural_state`**: Declares registers, register files, address spaces, aliases, constants
- **Functions**: Helper functions (C-like syntax), optionally `extern`
- **Always blocks**: Execute every cycle in parallel with instructions
- **Instructions**: Encoding + behavior definitions

### Types

Arbitrary-precision signed/unsigned integers: `signed<N>`, `unsigned<N>`. Aliases: `int` = `signed<32>`, `bool` = `unsigned<1>`, etc. Type conversions never implicitly lose precision or sign info; narrowing requires explicit casts.

### Instruction Format

```
InstructionName {
    encoding: field[msb:lsb] :: literal :: field[msb:lsb] :: ... ;
    behavior: { /* C-like statements */ }
}
```

Encoding fields are concatenated MSB-to-LSB with `::`. Literals use Verilog-style notation (`7'b0110011`, `3'b010`). Non-contiguous field ranges are supported.

### Key Operators

| Category | Operators | Notes |
|---|---|---|
| Arithmetic | `+`, `-`, `*`, `/`, `%` | Result widths expand to avoid overflow |
| Bitwise | `&`, `\|`, `^` | Result width = max(operands); mixed sign → signed |
| Shift | `<<`, `>>` | Preserves LHS type; signed → arithmetic shift |
| Concat | `::` | Result always unsigned, width = sum of operands |
| Extract | `val[i]`, `val[hi:lo]` | Bit or range extraction |
| Compare | `==`, `!=`, `<`, `>`, `<=`, `>=` | Returns `unsigned<1>` |
| Cast | `(signed)`, `(unsigned)` | Change signedness, preserve width |

### Spawn Blocks

`spawn { ... }` at the end of instruction behavior marks decoupled (asynchronous) execution. Lifts program order constraints — subsequent instructions can issue while spawn executes.

## CoreDSL MLIR Dialect (ShortNail)

The CoreDSL dialect (`coredsl::`) is defined in ShortNail (`deps/shortnail/`). It represents CoreDSL constructs as MLIR ops, using CIRCT's `hwarith` types (`ui<N>`, `si<N>`).

### Op Hierarchy

```
builtin.module
  └── coredsl.isax "NAME" {           (ISAXOp: SymbolTable, IsolatedFromAbove)
        ├── coredsl.register ...       (RegisterOp: core_x|core_pc|core_fp|local)
        ├── coredsl.addrspace ...      (AddressSpaceOp: core_mem|core_csr)
        ├── coredsl.alias ...          (AliasOp)
        ├── func.func @helper { ... }
        ├── coredsl.always @name { ... coredsl.end }  (AlwaysOp)
        └── coredsl.instruction @name(encoding...) {   (InstructionOp)
              ├── coredsl.get / coredsl.set
              ├── coredsl.bitextract / coredsl.bitset
              ├── coredsl.cast / coredsl.concat
              ├── coredsl.or / coredsl.xor / coredsl.and
              ├── coredsl.shift_left / coredsl.shift_right / coredsl.mod
              ├── hwarith.add / hwarith.sub / hwarith.mul / hwarith.div / ...
              ├── func.call @helper(...)
              ├── coredsl.spawn { ... coredsl.end }  (SpawnOp)
              └── coredsl.end                         (EndOp)
```

### Instruction Encoding

`InstructionOp` stores encoding as `StrArrayAttr` — array of string elements from MSB to LSB:
- Literal patterns: `"0110011"`, `"010"` (binary strings)
- Operand fields: SSA values with types like `%rs1 : ui5`, `%imm : ui12`

Key accessors:
- `getEncodingMask()` → `StringAttr` with `0`/`1` for fixed bits, `-` for operand bits (e.g. `"0000010----------111-----1111011"`)
- `getEncodingFields()` → LSB/MSB positions of each operand field
- `getName()` → instruction symbol name

### Treenail Encoding Field Attributes

Treenail (CoreDSL → MLIR compiler) emits a discardable attribute `lil.enc_immediates` on each `InstructionOp`. This attribute maps encoding field SSA values back to their original CoreDSL field names and bit ranges.

**Attribute format** (array of arrays on `InstructionOp`):
```
lil.enc_immediates = [
  ["%TREENAIL_WAS_HERE_rs1_4_0", 4, 0, 0, "rs1"],
  ["%TREENAIL_WAS_HERE_imm7_6_0", 6, 0, 0, "imm7"]
]
```

Each element: `[ssa_name, start_bit, end_bit, reversed, clean_name]`

| Index | Field | Description |
|-------|-------|-------------|
| 0 | `ssa_name` | SSA value name in MLIR (with `TREENAIL_WAS_HERE_` prefix) |
| 1 | `start_bit` | MSB of the field within the encoding segment |
| 2 | `end_bit` | LSB of the field within the encoding segment |
| 3 | `reversed` | `1` if the bit order is reversed relative to CoreDSL declaration |
| 4 | `clean_name` | Original CoreDSL field name (e.g., `rs1`, `imm7`) |

**Field naming convention**: Treenail prefixes encoding field SSA values with `TREENAIL_WAS_HERE_{name}_{msb}_{lsb}` (e.g., `%TREENAIL_WAS_HERE_rs1_4_0` for a 5-bit `rs1` field). Non-contiguous fields are split into multiple SSA values with the same base name but different bit ranges (e.g., `%TREENAIL_WAS_HERE_imm12_11_5` and `%TREENAIL_WAS_HERE_imm12_4_1`).

**Bit reversal**: Treenail detects reversed encoding fields (`start < end` in CoreDSL) and sets `reversed=1`. The reversal is resolved at MLIR generation time via `coredsl.bitextract` ops with reversed indices — downstream passes see the field in its natural bit order. The `reversed` flag in the attribute is informational only.

**Consumer**: `AnalyzeISAX.cpp` (in `deps/longnail/shortnail/lib/Conversion/AnalyzeISAX/`) requires this attribute on every `InstructionOp`. It reads the 5th element (`clean_name`) to map encoding arguments to clean field names for YAML output. The pass asserts if the attribute is missing — MLIR must be produced by a Treenail version that includes the clean name element.

**Source**: `deps/treenail/app/src/main/java/de/tudarmstadt/esa/treenail/codegen/EncodingFieldSwitch.java` (line ~121).

### Register Modes

| Mode | Keyword | SCAIE-V | Examples |
|---|---|---|---|
| `core_x` | `coredsl.register core_x @X[32] : ui32` | RdRS1/RdRS2/WrRD | GPR file |
| `core_pc` | `coredsl.register core_pc @PC : ui32` | RdPC/WrPC | Program counter |
| `core_fp` | `coredsl.register core_fp @F[32] : ui32` | (reserved) | FP regs |
| `local` | `coredsl.register local @REG : ui14` | Custom reg interfaces | ISAX-private state |

### splitop-gen (split-datapath operator SV generator)

Python tool at `deps/splitop-gen/` that consumes the YAMLs emitted by `-export-splitop-yaml` and produces SystemVerilog modules matching the `hw.module.extern` declarations produced by LILToHW. Invoked automatically by `splitop.py` after `run_longnail`; outputs (plus the hand-written `rtl/*.sv` primitives) are appended to `filelist.f` so SCAIE-V picks them up.

Runs only when `LN_ENABLE_DATAPATH_SPLIT=y` (which itself requires `LN_SCHED_ALGO_MI=y`).

## Architecture

### Main Orchestrator (`dispatch.py`)

The central state machine that:
1. Loads `.config` via `kconfiglib`
2. Resolves entry point and MLIR paths (`entrypoint.py`)
3. Runs an iterative refinement loop: Longnail HLS → SCAIE-V integration → simulation → synthesis plugins → critical path feedback → re-optimize

The loop terminates when no more critical chains are found from synthesis.

### Entry Points (`entrypoint.py`)

Six mutually exclusive entry points control where the pipeline starts:
- `DEFAULT_ENTRY_POINT` — Full flow from CoreDSL via Treenail
- `COREDSL_MLIR_ENTRY_POINT` / `PREPARED_MLIR_ENTRY_POINT` / `SOLUTION_MLIR_ENTRY_POINT` — Pre-compiled MLIR
- `SV_ENTRY_POINT` — Direct SystemVerilog + ISAX YAML
- `NO_ISAX_ENTRY_POINT` — Bypass ISAX integration entirely

### Pipeline Stages

| Module | Role |
|---|---|
| `treenail.py` | CoreDSL → MLIR translation (parallelized via ThreadPoolExecutor) |
| `longnail.py` | HLS scheduling (MLIR → SystemVerilog), supports multiple ILP solvers |
| `scaiev.py` | Integrates custom functional units into RISC-V cores |
| `simulation.py` | Runs cocotb testbenches with ELF→HEX, optional Renode ISS lockstep |
| `toolchain.py` | Builds the dynamic-ISAX LLVM fork (`deps/llvm_dynamic_isax`) and drives testbench compilation/disassembly with `-misax-desc=<analysis.yaml>` |
| `picolibc.py` | Compiles minimal C library for baremetal targets |
| `renode.py` | Generates Renode emulator configuration |

### Core Support Framework (`cores/`)

Each supported RISC-V core (CVA5, CVA6, VexRiscv, NaxRiscv, Orca, Piccolo, PicoRV32) has a Python module implementing the `CoreSupport` interface registered via `scaiev.py`. Core selection happens through Kconfig.

### Configuration System

Uses `kconfiglib` (Python Kconfig implementation). Root `Kconfig` sources:
- `build/cores_Kconfig` and `build/ISAX/Kconfig` — auto-generated by `gen_config.py`
- `configs/LN_Kconfig` — Longnail HLS settings (cell library, clock period, ILP solver, scheduling algorithm)
- `configs/sim_Kconfig` — Simulation options (cycle limits, ISS lockstep, verbosity)
- `configs/SCAIE-V_Kconfig` and `configs/advanced_Kconfig`

### ISAX Definitions (`isaxes/`)

Each ISAX lives in a subdirectory with a `.core_desc` (CoreDSL) file and a `paths.csv` mapping Kconfig symbols to MLIR files. See `isaxes/README.md` for the encoding reference table.

### Plugin System (`plugins/`)

Rudimentary plugin architecture. `dispatch.py` walks `plugins/*/` looking for files matching the entry point name (e.g., `synthesis_plugin.py`). LibreLane is the reference synthesis plugin, returning critical path chains for iterative optimization.

### Error Codes (`error.py`)

Stratified by component: USER_ERROR=1, INTERNAL_ERROR=2, TN=50+, LN=100+, SCAIEV=150+, PICOLIBC=190+, SIM=200+, LLVM=220+, LIBRELANE=230+.

### Test Infrastructure

- `custom_tbs/` — Assembly/C++ test sources with `*_expected.txt` output files
- `sim/` — cocotb test runners (`processortest.py`, `test_iss_lockstep.py`, `test_default.py`), bus models, ISS, trace utilities
- `tools/run_integration_tests.py` — CLI for running integration tests across core/ISAX combinations

### Dependencies (`deps/`)

Git submodules: `shortnail` (CoreDSL MLIR dialect; also vendored under `longnail/shortnail/` where `AnalyzeISAX` lives), `llvm_dynamic_isax` (LLVM 21 fork with native dynamic-ISAX support — consumes the `analyze-isax` YAML via `-misax-desc`), `scaie-v` (core integration framework), `treenail` (CoreDSL compiler). Patches in `patches/` are applied to these dependencies.

### Output

Results go to numbered directories under `outputs/` (e.g., `outputs/run_0/`). Each run copies its `.config` for reproducibility. Override with `OUTPUT_PATH` env var.

### Testing

- The full integration test suite takes a very long time. During development, **run failing tests individually** via `make ci` with the appropriate environment variables instead of rerunning the entire suite.
- Example individual test: `CORE="CVA5" ISAXES="SBOX" SIM_ENABLE="y" TB_PATH="custom_tbs/sbox.cpp" TB_EXPECTED_PATH="custom_tbs/sbox_expected.txt" make ci`
- Use `./tools/run_integration_tests.py --show-results <folder>` to view results from a previous test run.

### Environment

Development runs inside a Debian-based Docker container. Additional tools (e.g., waveform viewers like GTKWave for VCD analysis) can be installed via `apt install`.
